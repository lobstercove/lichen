### Fix 1: Replicate Genesis Auto-Fund on Joining Nodes (F-01)

**File:** `validator/src/main.rs` (~line 7596, after step 5 genesis account reconstruction)

**What:** Add the same 10K LICN auto-fund from treasury to genesis wallet that the genesis producer does.

**Code change:**
```rust
// After: state_for_blocks.put_account(&gpk, &genesis_account).ok();
// Add:

// 5b. Auto-fund genesis wallet from treasury (matches genesis producer)
let ops_fund_spores = Account::licn_to_spores(10_000);
if let Some((_, treasury_pk, _, _)) = dist_recipients.iter().find(|(r, _, _, _)| r == "validator_rewards") {
    if let Ok(Some(mut treasury_acct)) = state_for_blocks.get_account(treasury_pk) {
        if treasury_acct.deduct_spendable(ops_fund_spores).is_ok() {
            genesis_account.add_spendable(ops_fund_spores).ok();
            state_for_blocks.put_account(treasury_pk, &treasury_acct).ok();
            state_for_blocks.put_account(&gpk, &genesis_account).ok();
            info!("  ✓ 📡 [sync] Auto-funded genesis wallet with 10,000 LICN from treasury");
        }
    }
}
```

**Verification:** After fix, check EU genesis wallet balance via RPC: should be 10,000 LICN.

### Fix 3: Remove Ghost Validator Purge (F-04)

**File:** `validator/src/main.rs` (~lines 6283-6297)

**What:** Comment out or remove the ghost validator purge entirely. This is a non-deterministic direct state write that can cause different nodes to have different treasury balances and validator sets.

If ghost cleanup is needed, it should be a consensus-level transaction (like opcode 28 or similar) that all nodes process identically.

**Code change:** Remove or gate behind a flag that's disabled by default.

**Verification:** All 3 nodes should produce identical treasury balances after genesis.

## Phase 2 — FIX BEFORE PRODUCTION (Fixes F-03, F-07, F-09, F-10)

These improve resilience but won't cause immediate state divergence under normal operation.

### Fix 5: Add Idempotency Guard at Top of apply_block_effects (F-03)

**File:** `validator/src/main.rs` line ~3152

**What:** Add an early return if `reward_distribution_hash(slot)` already exists. This becomes the master guard: if effects were already applied for this slot, skip everything.

```rust
// Right after the slot==0 check:
if !skip_rewards {
    if let Ok(Some(_)) = state.get_reward_distribution_hash(block.header.slot) {
        return; // Effects already applied for this slot
    }
}
```

This makes apply_block_effects truly idempotent — if we crash between put_block_atomic and apply_block_effects, retry on restart will succeed.

---

### Fix 6: Add Explicit Parent Request for Orphaned Blocks (F-07)

**File:** Sync manager

**What:** When a block arrives whose parent is unknown, immediately request the parent block from the sender peer rather than waiting for the next 5-second sync probe.

---

### Fix 7: Deterministic Bootstrap Replication (F-09)

**File:** `validator/src/main.rs` lines 7978-8009

**What:** Instead of conditionally replicating bootstrap grants based on whether the pool entry is missing at processing time, make the bootstrap replication part of the genesis block processing (step 6b already handles this for slot-0 validators). Ensure ALL validators registered in the genesis block get their bootstrap grants in step 6b regardless of timing.

---

### Fix 8: Two-Phase Fork Adoption (F-10)

**What:** Write new block to a staging area first, then atomic swap with old block. If crash during staging, old block remains valid. This is a significant refactor — defer to post-launch.

## Phase 3 — FIX BEFORE MAINNET (Fixes F-06, F-11, F-13, F-14)

### Fix 9: Add P2P Message Handlers (F-06)
Add match arms for all unhandled message types. At minimum, log them.

### Fix 10: WebSocket Backpressure (F-11)
Replace unbounded channels with bounded channels. Drop messages for slow clients.

### Fix 11: Gate Admin RPC Endpoints (F-13)
Add `--enable-admin-rpc` flag. Disable by default. Require confirmation for state-modifying admin calls.

### Fix 12: Cache Orderbook (F-14)
Maintain in-memory orderbook that updates incrementally on each new order/cancel.
